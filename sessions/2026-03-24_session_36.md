# Session 2026-03-24 — Sentiment Analysis PoC (MyProjects)

## What we did

### Design & Spec
- Full brainstorming cycle for a new project: sentiment analysis model for an AI-based job search platform
- Approved architecture: hybrid real-time classifier + LLM synthesis engine
- Key design decisions:
  - Dual output: JSON profile (AI/scrapers) + Markdown coaching report (humans)
  - Explicit match feedback layer (structured rating: good/moderate/fair/inadequate/related + free text)
  - Hybrid persistence: session signals (short-term) + cumulative candidate profile (long-term)
  - Skill profiler: inferred level vs claimed level, with evidence and gap notes
  - Job role taxonomy: maps role types to required skills (management → leadership/comms, etc.)
  - Application tailoring engine: per-role match score, highlight skills, gap actions, application angle
- Spec written to `docs/superpowers/specs/2026-03-24-sentiment-analysis-jobsearch-design.md` (committed)
- Project scaffold created at `my-projects/sentiment-jobsearch/README.md`

### PoC Implementation (v2)
- Built `my-projects/sentiment-jobsearch/poc_sentiment.py` — full pipeline in one file:
  - Real-time classifier via Groq (llama-3.3-70b) with rule-based emotion tag derivation
  - Session signal store, match feedback parser, skill profiler, tailoring engine, LLM synthesis
  - Output: JSON profile + Markdown report per candidate

## Lessons learned
- Groq API requires `User-Agent: DQIII8/1.0` header — without it, returns 403
- LLM synthesis prompts must be very explicit about JSON string escaping (`\\n` for newlines in markdown_report) to avoid parse failures
- Rate limiting: 20s cooldown between candidates is sufficient for Groq free tier at this call volume
- Skill profiler authenticity score is a strong signal: Col. Hayes 0.95 (claims match demonstrated style) vs Sara Chen 0.60 (bootcamp claims outrun demonstrated depth)
- Domain context detection via keywords is fast and sufficient for routing at PoC stage; replace with classifier for production
